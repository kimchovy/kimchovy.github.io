## 들어가기에 앞서 : HTTP 메서드(Methed)란?

HTTP 메서드란 클라이언트와 서버간에 요청(Request)과 응답(Response) 데이터를 전송하는 방식을 뜻해요.

클라이언트가 서버측으로 보내는 데이터가 '요청', 그리고 해당 요청에 대해 서버가 클라이언트측으로 보내는 데이터를 '응답'이라고 합니다.

`HTTP 메서드에는 주로 쓰이는 주요 메서드 5가지와 기타 메서드 4가지로 총 9가지가 있습니다. 함께 알아볼까요?`

## 주요 메서드 5가지

GET
: 리소스 조회

POST
: 요청 데이터 처리, 주로 등록에 사용

PUT
: 리소스를 대체(덮어쓰기), 해당 리소스가 없으면 생성

PATCH
: 리소스 부분 변경

DELETE
: 리소스 삭제

### 1. GET 메서드

- 리소스를 조회할때 사용하는 메서드에요.
- 서버측으로 전달하고 싶은 데이터는 쿼리(query)를 통해 전달합니다.
    > 쿼리(query)는 query string 또는 query parameter라고도 불립니다.
    >> 쿼리 외에 메시지 바디를 사용해서 데이터 전달도 가능하나, 지원하지 않는 곳이 많아서 권장하지 않아요.
- key=value 형태로 구성되어 있으며, '?'로 시작하고 값의 추가시에는 '&'를 붙이면 됩니다.
    > 쿼리(query)의 뜻 자체가 '질문' 또는 '물음표' 라는 것을 생각해보면 뜬금없이 '?'가 나오는것이 어찌보면 당연하다고 볼 수 있어요!
- 조회시에 GET 메서드가 아닌 POST 메서드도 사용은 가능하지만, GET 메서드는 캐싱이 가능하기에 GET을 사용하는 것이 유리하다고 합니다.
- 데이터가 URI상에 그대로 노출되기 때문에 보안성에 취약합니다.

#### 간단한 부연설명 GET

- 사실 예시라고 할것도 없이 우리가 일상에서 웹서핑을 하는 과정 자체가 매순간마다 어느 서버로 GET요청을 날리고 있고, 해당 요청에 대한 서버의 응답으로 다양한 데이터들을 받아와서 우리의 모니터에 보여지게 되는 것이에요.
    > 아직까지 감이 오지 않는다면 제가 좋아하는 네이버 웹툰 중 하나로 예를 들어 보여드릴게요!

- 다음은 네이버 웹툰 '킬러 배드로'의 URI입니다.(글 작성일 기준 연재중)
    - `https://comic.naver.com/webtoon/list?titleId=816798&tab=sat`
    - 위 링크를 보면 쿼리가 어디부터 시작되는지, 쿼리가 무엇을 의미하는지 대략적으로 알 수 있죠.
    > 앞부분인 `https://comic.naver.com/webtoon/list` 를 제외하면 `?titleId=816798&tab=sat` 만 남게 되는데, '?' 로 시작해서 중간에 '&' 기호를 제외한 `titleId=816798` , `tab=sat` 두개의 key=value 형식이 쿼리 파라미터에 담긴 것이에요.
    >> 위에서 언급했듯이 해당 '킬러 배드로' 라는 웹툰의 ID값이 서버 내부적으로 'titleId=`816798`' 라는 것을 누구나 파악할 수 있기에 보안성에 취약할 수 있다는 거죠.
    >>> 앞으로 어떤 사이트든 접속하게 되면 사이트 주소를 중간중간 한번씩 확인해보세요!(생각보다 다양한 정보와 옵션들이 보일거에요)

### 2. POST 메서드

- 서버에 데이터를 전송해서 새로운 리소스를 생성하거나 특정 프로세스를 처리하는 메서드에요.
- 메시지 바디를 통해 서버로 요청 데이터를 전달하죠.
    > 요즘은 JSON이 거의 표준이라고 합니다.
- '애매하다 싶으면 POST를 사용하라'는 말이 있을 정도로 다양하게 활용되는 메서드에요.
    > 그만큼 정해진 틀이 없기에 POST 요청별로 해당 요청 데이터를 처리하는 방법을 리소스마다 정의해줘야 합니다.

#### 간단한 부연설명 POST

- 쉽게 생각하자면, 우리가 인터넷상에서 무언가를 남기고 생성하는 거의 모든 행위가 POST라고 이해하면 쉬워요.(모든 경우에 해당되는건 아니지만)
    - 이를테면 어딘가에 회원가입을 한다거나(계정 생성), 게시글 또는 댓글을 작성, 온라인 결제, 파일 업로드 등의 행위는 대부분 POST로 이루어집니다!

### 3. PUT 메서드

- 리소스를 대체(수정)하는 메서드에요.
- 만약 클라이언트의 해당 요청에 대한 데이터가 이미 서버에 존재한다면 대체(덮어쓰기)하고, 없다면 새로운 리소스를 생성해요.
    > 새로운 리소스를 생성하는 POST 메서드와의 차이점은 클라이언트가 리소스의 URI를 알고 지정해서 보내느냐의 차이에요.
- 만일 PUT 요청시 빠진 필드가 있다면 해당 필드 없이 리소스가 생성 또는 대체(덮어쓰기)되므로 주의해야 해요!
    > 따라서 리소스의 특정 부분만 변경하길 원할경우 후술할 PATCH 메서드를 사용하면 됩니다.

#### 간단한 부연설명 PUT

- PATCH의 부연설명에서 비교하며 알아볼게요!

### 4. PATCH 메서드

- 서버의 리소스를 일부만 업데이트하는 메서드에요.
- PUT과는 달리 Message Body에 업데이트 하고자 하는 필드만 담아서 요청하면 되죠.
    > 만일 PATCH를 지원하지 않는 서버에서는 POST를 대신 사용할 수 있다고 합니다!

#### 간단한 부연설명 PATCH

- PUT 메서드와 비교해서 설명을 하려고 하는데 쉬운 예시를 생각해보니, 공책이 적절할 것 같아서 이것으로 예를 들어볼게요.(정확한 예시는 아니고, 비유 정도로만 생각하면 됩니다)
    - 공책에 연필로 마구 낙서도 하고 이것저것 글도 썼다고 가정해볼까요?
    > 이 공책은 마법공책이라 딱 '1권'만 존재해야하고 만일 2권이 되는 순간, 앞서 있던 공책의 자리는 나중에 생긴 공책이 그 자리를 차지한다고 해볼게요.
    - 근데 어느날 공책이 너무 지저분해 보여서 어느 부분을 수정하고 싶어졌다고 하면 지우개로 지우고 다시 쓰거나 맘에 들지 않는 부분만 지우면 될 일이죠.(`PATCH`)
    - 생각해보니 전체 내용이 맘에 안들어서 일일이 지우며 수정하는 방법보다는 새 공책에 처음부터 작성하는게 나을거라는 생각이 들어서 기존 공책을 버리고 새 공책으로 대체하여 사용하기로 했어요.(`PUT`)
    > 부연의 부연설명으로 단순히 결과만 놓고보면 `PUT` 메서드를 사용하는게 `DELETE`, `POST`를 순차적으로 하는것과 무슨 차이지? 싶을 수 있는데요.
    >> `PUT` 메서드는 리소스를 갱신하는 과정에서 `삭제` 후 `재생성`이라는 동작을 하지 않고, `덮어쓰기(Overwrite)`하는 방식으로 동작할 가능성이 커요.
    >>> 따라서 실제 `삭제` 후 `생성` 하는 `DELETE`, `POST` 과정은 `DB(Database)` 또는 `캐시(Cache)`에서 해당 리소스의 ID값이 바뀌거나, 어떠한 트리거가 작동할 가능성이 있다는 차이점이 있고, 만일 `DELETE`에는 성공했으나 `POST`요청이 실패한다면 그냥 리소스만 날려버린 결과가 되는 것이죠!

### 5. DELETE 메서드

- 요청한 리소스를 삭제하는 메서드에요.
- 데이터 삭제에 성공할 경우, body값 없이 성공 응답만 보내죠.
    - 만일 존재하지 않는 리소스에 대한 요청이거나 해당 권한이 없을 경우 서버에서 정의된 응답메시지를 받게돼요.
    > 이게 전부에요!



## 기타 메서드 4가지

HEAD
: GET과 동일하지만 메시지 부분(body 부분)을 제외하고, 상태 줄과 헤더만 반환

OPTIONS
: 대상 리소스에 대한 통신 가능 옵션(메서드)을 설명(주로 CORS에서 사용)

CONNECT
: 대상 자원으로 식별되는 서버에 대한 터널을 설정

TRACE
: 대상 리소스에 대한 경로를 따라 메시지 루프백 테스트를 수행

### 6. HEAD 메서드

- GET 메서드와 요청 방식이 비슷하지만, 실제 응답 본문(Body)을 포함하지 않고 클라이언트가 요청한 URI에 대한 응답 헤더만 전달하는 메서드에요.(만일 본문(body부분)이 존재하더라도 무시해야 한다고 해요!)
- 이 메서드는 요청에 대한 응답값(컨텐츠 또는 데이터)이 서버에 존재하는지 여부를 확인하는데에 쓰여요.

### 7. OPTIONS 메서드

- 서버에서 지원하는 HTTP 요청 방식을 확인할때 사용하는 메서드에요.
- 클라이언트가 OPTIONS 메서드와 URI를 서버로 전달하면, 서버는 해당 서버가 지원하는 HTTP 요청 방식(GET, POST, PUT 등)을 응답 헤더에 포함하여 클라이언트에게 응답하죠.
    > 이 요청은 실제 데이터가 아닌, 응답 서버가 지원하는 메서드와 어떤 권한을 가졌는지 파악하는데 사용됩니다!

### 8. CONNECT 메서드

- 클라이언트가 요청한 리소스에 대해 양방향 연결을 시작하는 메서드에요.
- 예를 들면, SSL(HTTPS)를 사용하는 웹사이트에 접속하는데 사용되죠.
    > SSL(HTTPS) 또한 굉장히 중요한 개념이니 검색해서 알아보시길 추천드려요.
    >> (참고: SSL과 HTTPS는 완전히 동일한 개념이 아니니 꼭!! 검색해서 이게 각각 무엇을 뜻하는 용어인지 알고 넘어가길 바랄게요)

### 9. TRACE 메서드

- 클라이언트로부터 서버까지의 네트워크 경로를 체크하는 메서드에요.
- 클라이언트와 서버 사이에 중계 역할을 하는 중간 서버의 존재를 체크하는데에 쓰이죠.
    > 여기서 어디선가 들어봤을법한 용어인 '프록시 서버(Proxy Server)'가 바로 '중간 서버'를 뜻하는 용어에요!
    >> 이 프록시 서버는 장치가 될수도, 응용 프로그램이 될수도 있으며, 프록시 서버가 사용되는 위치에 따라 'Forward Proxy'와 'Reverse Proxy'로 구분됩니다.
- 여기서 생길만한 궁금증이 하나 있는데, 어째서 서버와 클라이언트가 직접 연결되는게 아니라 다리를 건너건너 데이터를 교환하는지, 이게 더 비효율적이라는 생각이 들수도 있습니다.
    > 물론, 건너건너 데이터를 주고 받게 되니 중간에 패킷 로스(데이터 손실) 또는 인터넷 연결 속도가 느려질수도 있다는 단점이 있지만, 보안성과 캐싱에 장점이.. 적다보니 너무 길어지는듯 해서 이것도 추후 글을 따로 작성해야겠네요...
- 간단하게 비유를 해보자면, 우리들을 클라이언트, 전기를 만드는 여러 발전소가 서버라고 생각해볼까요? 내가 전기가 필요하다고 발전소에 요청을 한다고해서 발전소에서 우리집으로 직접 전기를 보내주는 것이 아니라 송전탑, 변전소, 변압기 등등을 거쳐서 우리에게 도달하는 과정을 생각해보면 송전탑, 변전소, 변압기 등등이 '프록시 서버'의 역할을 하는 것이죠!
